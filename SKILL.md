# Code Review

## 审查流程

按以下顺序执行审查，每一步独立输出结论：

1. **理解上下文** — 确认代码的目的、所属模块、涉及的语言和框架
2. **正确性检查** — 找出逻辑错误、边界条件遗漏、空值/异常未处理
3. **安全性检查** — 按 OWASP Top 10 扫描：注入、认证绕过、敏感数据泄露、XSS、不安全的反序列化等
4. **性能检查** — 找出 O(n²) 循环、不必要的内存分配、缺失索引、N+1 查询、阻塞调用
5. **可维护性检查** — 命名是否清晰、函数是否过长（>50 行警惕）、职责是否单一、重复代码
6. **输出审查报告**

## 报告格式

对每个发现的问题使用以下格式：

```
### [严重程度] 问题标题
- **位置**: 文件名:行号
- **问题**: 一句话描述问题
- **原因**: 为什么这是个问题
- **建议**: 给出具体的修改代码（不是泛泛的建议）
```

严重程度分三级：
- 🔴 **必须修复** — bug、安全漏洞、数据丢失风险
- 🟡 **建议修复** — 性能问题、可维护性差、不符合惯例
- 🟢 **可选优化** — 代码风格、微小改进

## 审查结束时

在报告末尾给出一行总结：共发现 X 个必须修复、Y 个建议修复、Z 个可选优化。

如果代码整体质量良好，明确说出做得好的地方（一句话即可）。

## 约束：不要做的事

- 不要给出无法直接执行的模糊建议（如"提高代码质量"、"优化性能"）。每条建议必须附带具体修改代码
- 不要审查与当前变更无关的已有代码，除非它直接导致问题
- 不要因为个人风格偏好否定代码（如大括号换行 vs 同行、tab vs space）——除非项目有明确的 lint 规则
- 不要在没有证据的情况下猜测性能问题。只指出可论证的性能瓶颈
- 不要一次输出超过 10 个问题。如果超过，按严重程度保留前 10 个，其余归入"其他发现"列表（仅标题）
- 不要添加"辛苦了"、"代码写得不错但..."等社交客套。直接说发现
